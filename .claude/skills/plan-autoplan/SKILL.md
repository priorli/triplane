# /plan-autoplan — run the five-role plan review end-to-end

You are the orchestrator. Your job is to chain the five planning reviewers in a fixed order, make sure each one produces exactly one section in the shared `PLAN_REVIEW.md`, and then close the file with a `## Next steps` block that gives the user a single, dedupled action list plus a recommended next command.

You do not write review content yourself. Each reviewer skill has its own voice, rubric, and invariants — invoke them, don't replace them. Your only independent writing is:
- The file header (H1 + provenance stub) at the start.
- The `## Next steps` block at the end.
- An `## Errors` block **only** if a reviewer fails.

## Invariants

1. **Fixed order, no reshuffling.** CEO → Engineering → Design → DevEx → QA. Each reviewer depends on the ones before it (e.g., Eng plans against CEO's post-cut backlog; QA plans against Eng's risk list). If you run them out of order, their cross-references break.
2. **Stop on first error.** If any reviewer fails — the skill errors, the file write fails, or the review section is empty — halt, write an `## Errors` section to `PLAN_REVIEW.md` naming the reviewer and the error, and print a one-line failure report. Do not "skip and continue." A broken chain is a user-visible problem, not a papered-over one.
3. **Never regenerate `IDEA.md`.** That's `/ideate`'s job. If the brief needs changes, bail out and point at `/ideate`.
4. **Never draft spec files.** No writes to `specs/**`. That's `/feature add`.
5. **Never run `bin/init.sh` or `rewrite-docs.sh`.** That's `/init-app`. You are strictly *upstream* of the bootstrap — the Triplane Forge worker runs you first, then runs `/init-app` in a separate agent session after your output is on disk.
6. **Idempotent on re-run.** If `PLAN_REVIEW.md` already exists with reviewer sections in it, ask the user: **overwrite** (start fresh), **append a new round** (write a second pass with a datestamped H2 divider), or **abort** (leave the existing file and halt). Match the overwrite/append/abort pattern `/ideate` uses at Step 1.
7. **One output file.** `PLAN_REVIEW.md` at the worktree/repo root. Nothing else.
8. **Do not print the full file back to chat.** It's already on disk. Print a short summary at the end — filename, per-reviewer scores, next command.

## Step 1 — Locate `IDEA.md` and decide the write mode

1. Look for `IDEA.md` at the worktree/repo root. If it's missing, halt with a one-line pointer at `/ideate`. Do not invent a brief.
2. Look for `PLAN_REVIEW.md` at the same root.
   - **If it doesn't exist:** proceed to Step 2 (fresh write mode).
   - **If it exists and has no reviewer sections** (e.g., only a provenance stub left over from a crashed earlier run): proceed to Step 2 and overwrite.
   - **If it exists with at least one reviewer section:** ask the user, in a single message: *"`PLAN_REVIEW.md` already has reviewer sections. Overwrite (start fresh), append a new round (datestamped), or abort?"*. Wait for explicit `overwrite`, `append`, or `abort`. Do not proceed on a soft answer like "sure" or "looks good" — re-ask if ambiguous.

## Step 2 — Create the `PLAN_REVIEW.md` header

Write (or, in append mode, append) this header to `PLAN_REVIEW.md`. Use Write for overwrite mode, Edit for append mode.

**Fresh / overwrite mode:**

```markdown
# Plan review — <product name from IDEA.md>

_Generated by `/plan-autoplan` on <YYYY-MM-DD> against `IDEA.md`. Five reviewers: CEO, Engineering, Design, DevEx, QA. Each section below is the output of the corresponding skill._
```

**Append mode:**

```markdown

---

# Plan review — round N (<YYYY-MM-DD>)

_Second-pass review against the same `IDEA.md`. Previous review round is above._
```

Where `<YYYY-MM-DD>` is the real current date and `N` is the next round number (count the existing `# Plan review` headers in the file and add 1).

Do **not** write the `## Next steps` block yet. That comes after all reviewers are done.

## Step 3 — Invoke the five reviewers, in order

Invoke each reviewer by running its skill. Each reviewer is an invocable skill with its own SKILL.md and its own append-a-section contract. You do not need to pass arguments — they each read `IDEA.md` and `PLAN_REVIEW.md` from the worktree root.

Invoke them **one at a time**, waiting for each to finish (i.e., for its section to be present in `PLAN_REVIEW.md`) before starting the next. Parallel invocation would corrupt the file.

Run in this exact order:

1. **`/plan-ceo-review`** — scope and framing. Appends `## CEO review`. Score: PMF readiness.
2. **`/plan-eng-review`** — architecture and risks. Appends `## Engineering review`. Score: implementability. Depends on the CEO cut list.
3. **`/plan-design-review`** — rubric + interaction decisions + one cut. Appends `## Design review`. Score: design readiness. Depends on CEO cut list and Eng surface scope.
4. **`/plan-devex-review`** — onboarding friction + doc recommendations. Appends `## DevEx review`. Score: onboarding. Depends on the architecture and risk list.
5. **`/plan-qa-review`** — per-feature scenarios + cross-feature risks + regression watchlist. Appends `## QA review`. Score: testability. Depends on all four prior sections — runs last on purpose.

**After each reviewer:**

- Verify the reviewer's section was appended. If it's missing (e.g., the skill silently no-op'd), treat that as an error and go to Step 4's error path.
- Capture the reviewer's 0–10 score from the section body. You'll need them for the summary in Step 5.
- Do **not** summarize or commentate on the reviewer's output — their section stands on its own.

## Step 4 — Error handling

If any reviewer errors, fails to append, or leaves the file in a bad state:

1. Append an `## Errors` section to `PLAN_REVIEW.md` with one line naming the reviewer, the phase, and the error message. Example:

```markdown

## Errors

- **`/plan-eng-review` failed during execution.** Error: <one-line error message>. The chain halted after this reviewer. Prior reviewers' sections are preserved; subsequent reviewers did not run.
```

2. Halt. Do not run subsequent reviewers. Do not write the `## Next steps` block.
3. Print a one-line failure report to chat:

> plan-autoplan halted: `/plan-eng-review` failed (<error>). `PLAN_REVIEW.md` has `<list of completed sections>` and an `## Errors` block.

4. Return. The user is expected to re-run or fix the upstream issue.

## Step 5 — Write the `## Next steps` block

Once all five reviewers have appended their sections successfully, compose the closing block yourself. This is the only part of `PLAN_REVIEW.md` where you write original content.

Gather, from each reviewer's section:

- **CEO:** the cut list, the "what's missing" bullets.
- **Engineering:** the top 2 load-bearing risks, the gotcha watchlist.
- **Design:** the top 3 interaction decisions, the one-more-cut call.
- **DevEx:** the top 3 friction points, the README/CLAUDE.md recommendations.
- **QA:** the cross-feature risks, the regression watchlist.

De-dupe aggressively. If CEO cut a feature *and* Design reluctantly called for cutting the same one, that's one "cut X" action, not two. If Eng flagged the iOS ObjC-export gotcha *and* QA flagged it, that's one action, not two.

Then pick:

- **Top three concrete actions.** Actions users can take *before* or *during* the bootstrap. Not platitudes. "Sharpen the target user from X to Y" is an action. "Be careful about accessibility" is not.
- **One recommended first command.** Usually `/feature add <slug>` pointing at the first backlog item that survived all the cuts. Alternatives: `/ideate` (if multiple reviewers flagged the brief as too rough), or `/init-app` (if the forge is running you and the brief survived the review cleanly — let the forge worker continue with bootstrap).

Append this exact structure:

```markdown

## Next steps

**Top three actions:**

1. <one concrete action sentence — verb-first>
2. <another>
3. <another>

**Recommended first command:** `<command>` — <one-line reason>

**Per-reviewer scores:**

| Reviewer | Score |
|---|---|
| CEO (PMF readiness) | <N>/10 |
| Engineering (implementability) | <N>/10 |
| Design (readiness) | <N>/10 |
| DevEx (onboarding) | <N>/10 |
| QA (testability) | <N>/10 |
```

## Step 6 — Exit report

Print a **short** status to chat — three lines maximum:

> plan-autoplan complete.
> `PLAN_REVIEW.md` written with CEO/Eng/Design/DevEx/QA sections + Next steps.
> Top scores: CEO <N>, Eng <N>, Design <N>, DevEx <N>, QA <N>. Recommended next: `<command>`.

Do **not** paste the file content back to chat. The user can read the file directly.

## Files this skill touches

- **Reads:** `IDEA.md`, `PLAN_REVIEW.md` (to detect prior rounds)
- **Writes:** `PLAN_REVIEW.md` — header, delegates section-writing to the five reviewer skills, then writes `## Next steps`, with an optional `## Errors` block on failure
- **Never modifies:** `IDEA.md`, `CLAUDE.md`, `PLAN.md`, `README.md`, `specs/**`, `mobile_plan.md`, source files, or any configuration files
- **Never runs:** `bin/init.sh`, `./.claude/skills/init-app/rewrite-docs.sh`, `git commit`, `git push`, or any other side-effect commands

## Related skills

- `/plan-ceo-review`, `/plan-eng-review`, `/plan-design-review`, `/plan-devex-review`, `/plan-qa-review` — the five reviewers this skill chains.
- `/ideate` — upstream. Produces `IDEA.md`, which is this skill's input.
- `/init-app` — downstream. Consumes `IDEA.md` (and optionally takes a glance at `PLAN_REVIEW.md` for context) to bootstrap the downstream project. The Triplane Forge worker runs `/plan-autoplan` first, then `/init-app` in a separate agent session.
- `/feature add` — further downstream. Takes one item from the plan-reviewed backlog and drafts a spec file.
- `/release-check` — further downstream. The green-light verifier your Next steps section may recommend running after the first feature ships.

## When not to use this skill

- **`IDEA.md` does not exist.** Point at `/ideate`. Never invent a brief to review.
- **The user wants just one reviewer.** Invoke that reviewer directly — `/plan-ceo-review`, `/plan-eng-review`, etc. This skill is for the full chain only.
- **A spec already exists for the feature.** Plan-review is upstream of spec drafting. If the spec is done, run `/feature continue` instead.
- **The user wants to edit code, not plan.** This skill writes prose only. For implementation, the downstream commands are `/feature add` → `/scaffold` → `/feature continue`.
- **The user is running Triplane's forge on a *pristine demo* idea** where they want the fastest path to bootstrap. The plan review adds ~$1.50 and a couple of minutes. Skipping it is reasonable for template demos — that's why the forge UI makes it opt-in via a checkbox, not default-on.
